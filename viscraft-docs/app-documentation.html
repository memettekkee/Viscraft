Viscraft — Application Documentation

Complete user guide, feature reference, prompt system, error handling, and local setup instructions.

1. Introduction

Viscraft is an AI-powered product ad photography generator. It is designed for small business owners, marketers, and entrepreneurs who need professional-looking product photos without the cost of a photographer or studio.

The core workflow is simple: describe your product, choose a visual style (background, lighting, mood, camera angle, and props), preview the assembled prompt, and generate a studio-quality ad shot. The AI handles the rest. Results are saved in a persistent gallery organized by campaign, and any shot can be regenerated with a tweaked prompt at any time.

Primary Users

  • Small business owners launching products
  • E-commerce sellers needing product imagery
  • Marketers building ad creatives
  • Entrepreneurs testing product concepts

What It Is Not

  • A generic AI art generator
  • A photo editor or retouching tool
  • A social media scheduler
  • A video or animation tool

2. Feature List

  • User authentication — Register and login with email + password. JWT tokens persisted in localStorage for session continuity across page refreshes.
  • Campaign management — Create, list, and delete campaigns per user. Each campaign has a name, optional description, product category, and visual style.
  • AI-powered ad shot generation — Generate product ad shots using structured prompts via Pollinations.ai (kontext model).
  • Styling options system — 5 categories of visual styling options (background, lighting, mood, angle, props) fetched from the database. 20+ options available.
  • Generated prompt preview — The assembled AI prompt is shown to the user in real-time as they fill the form, before submitting.
  • Reference photo upload — Users can upload their own JPEG/PNG/WebP reference image (max 5MB) to guide generation in image-editing mode.
  • Ad shot gallery — All generated ad shots are displayed in a grid per campaign, ordered by creation sequence. Persists across page refreshes.
  • Regenerate flow — Clicking "Regenerate" on any ad shot re-opens the generate form with the original prompt pre-filled and the original image shown as a reference.
  • Detail modal — Clicking an ad shot opens a full-size image view with actions (regenerate, delete, download).
  • Async generation with real-time polling — A skeleton card appears immediately on submit; SWR polls every 3 seconds until the image is ready.
  • Failure states — ERR_03 (timeout), ERR_05 (bad AI response), ERR_13 (content policy) are all rendered visibly in the gallery as error cards.
  • Rate limiting — Maximum 5 generate requests per minute per user, enforced server-side.
  • Auto-seed sample campaign — New users get a sample campaign created automatically so the workspace is not empty on first login.
  • Onboarding tour — A 3-step guided tour walks first-time users through the key UI areas.
  • 404 page — Custom not-found page for unrecognized routes.
  • Daily log files — Backend writes structured logs to ./logs/app-YYYY-MM-DD.log alongside stdout output.

4. Prompt Building System

Raw product descriptions typed by users ("A bottle of cold brew coffee") produce inconsistent AI results on their own. The buildPrompt() function on the frontend assembles a structured prompt that reliably yields commercial-quality product photography.

How buildPrompt() Works

The function combines the user's product description with the promptValue of each selected styling option, separated into logical sections, and appends a fixed commercial photography suffix.

Example Input

FieldValue
Product DescriptionA bottle of cold brew coffee
BackgroundWhite Marble → white marble surface with subtle veining
LightingNatural Window → soft natural window light from left
MoodFresh & Clean → fresh clean aesthetic, bright and airy
Angle45 Degrees → 45-degree angle shot

Assembled Generated Prompt

Product: A bottle of cold brew coffee. white marble surface with subtle veining.
soft natural window light from left. fresh clean aesthetic, bright and airy.
45-degree angle shot. Commercial product photography, high resolution,
sharp focus on product, no watermark, no extra text.

Storage

Both the raw user input (user_prompt) and the assembled prompt (generated_prompt) are stored in the database. The generated_prompt is what actually gets sent to the Pollinations API. This separation preserves the user's original intent for the regenerate flow while giving the AI enough context to produce quality results.

Prompt Validation

  • Minimum length: 3 characters
  • Maximum length: 300 characters
  • Blocked words: nude, explicit, nsfw, gore (case-insensitive)
  • Validation runs server-side in sceneService.validatePrompt() — the frontend character counter is informational only

5. Error Handling

The application handles three categories of visible failure states, each rendered differently in the UI.

Generation Failure States (Async)

When a generation fails asynchronously, the skeleton polling card resolves into an error card in the gallery. The error card displays the failure reason and an option to retry.

Error CodeCauseUI Message
ERR_03Pollinations API did not respond within 60–90 seconds"Generation timed out. The AI took too long to respond."
ERR_05Pollinations returned an empty or unparseable response"Generation failed. The AI returned an unexpected response."
ERR_13Prompt was rejected by the AI content safety filter"Content policy violation. Please revise your prompt."

Form Validation Errors (Synchronous)

Validation errors from the server (ERR_04, ERR_11) are shown inline in the generate modal above the form fields. The submit button is also disabled client-side if the prompt is empty.

Rate Limit (HTTP 429)

When the rate limit is hit (ERR_02), the modal displays a dismissable error banner: "Too many requests. Please wait a moment before generating again."

Network / Auth Errors

The Axios JWT interceptor handles 401 responses globally — on token expiry, the user is logged out and redirected to the landing page. Generic network errors are shown via the ErrorBanner component at the top of the workspace.

6. Local Setup Instructions

Prerequisites: Go 1.21+, Node.js 18+, PostgreSQL (local or remote)

Backend Setup

cd viscraft-backend

# Copy and fill environment variables
cp .env.example .env

Edit .env with your values:

VariableDescription
DB_HOSTPostgreSQL host (e.g. localhost)
DB_PORTPostgreSQL port (default: 5432)
DB_NAMEDatabase name
DB_USERDatabase user
DB_PASSWORDDatabase password
JWT_SECRETRandom secret string for signing tokens
POLLINATIONS_API_KEYYour Pollinations.ai API key
APP_PORTServer port (default: 8089)
STORAGE_PATHPath for storing generated images (default: ./storage/images)
STORAGE_PUBLIC_URLPublic URL prefix for image files (e.g. http://localhost:8089/storage/images)
# Install Go dependencies
go mod download

# Run database migrations in order (001 through 005)
psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f migrations/001_init.sql
psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f migrations/002_add_used_reference_image.sql
psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f migrations/003_scenes.sql
psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f migrations/004_product_ad_pivot.sql
psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f migrations/005_drop_reference_scene_fkey.sql

# Start the server
go run main.go
# Server starts on http://localhost:8089

Frontend Setup

cd viscraft-frontend

# Install Node dependencies
npm install

# Copy and fill environment variables
cp .env.example .env

Edit .env:

VariableValue
VITE_API_BASE_URLhttp://localhost:8089
# Start the development server
npm run dev
# App runs on http://localhost:5173
Running in under 15 minutes: The only hard prerequisites are a running PostgreSQL instance and a Pollinations.ai API key (free tier available at pollinations.ai). Once the migrations are applied and both servers are running, the app is fully functional at http://localhost:5173.
