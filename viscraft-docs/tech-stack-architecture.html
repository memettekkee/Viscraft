Viscraft — Tech Stack & Architecture

Version 1.0  |  Go 1.21+ backend · React 19 frontend · PostgreSQL · Pollinations.ai

1. Overview

Viscraft is an AI-powered product ad photography generator. Users create campaigns, generate product ad shots by describing their product and selecting visual styling parameters, and manage a persistent gallery of results. All AI calls are routed through the Go backend — the browser never communicates with the AI API directly.

Core Problem Solved

Small business owners and marketers need professional product photos but can't afford a photographer. Viscraft generates studio-quality ad shots from a text description in under 60 seconds.

Key Architecture Properties

  • All AI calls through backend only
  • Async generation via Go goroutines
  • Images persisted to local filesystem
  • Per-user data isolation at DB layer
  • Stateless JWT authentication

5. Middleware Chain

Every request passes through a global middleware chain before reaching any handler. The generate endpoint has two additional middleware layers applied specifically to it.

CORS → RequestID → Timeout (30s) → Recovery → JWTAuth * → RateLimit * (generate only)

* JWTAuth is applied to all protected routes. RateLimit (5/min) is applied only to POST /scenes/generate.

MiddlewareResponsibility
CORSSets permissive CORS headers to allow frontend on any origin during development
RequestIDGenerates a UUID per request and attaches it to context. Included in all response envelopes for tracing.
Timeout30-second deadline for handler execution. Prevents slow requests from leaking goroutines.
RecoveryCatches panics and returns a 500 JSON response rather than crashing the server
JWTAuthParses and validates the Bearer token. Sets userId and email on the Gin context for downstream handlers.
RateLimitIn-memory per-user token bucket: max 5 generate requests per 60-second window

6. AI Integration Architecture

Two Generation Modes

The pkg/imagegen client automatically selects the correct Pollinations API endpoint based on whether a reference image is provided:

Text-to-Image (no reference)

Endpoint: GET /image/{prompt}

GET https://gen.pollinations.ai/image/{url-encoded-prompt}
  ?model=kontext&nologo=true
Authorization: Bearer {API_KEY}

Returns raw PNG bytes. Fast (10–30s), reliable.

Image Editing (with reference)

Endpoint: POST /v1/images/edits

POST https://gen.pollinations.ai/v1/images/edits
Content-Type: multipart/form-data

Fields: prompt, model=kontext
File:   image (reference PNG)

Slower (30–90s), ~20% success rate for complex edits.

Async Generation Lifecycle

1
HTTP handler receives POST /scenes/generate, validates JWT + prompt, verifies campaign ownership.
2
InsertProcessing() writes a DB record with status="processing" and returns the new sceneId.
3
Handler returns HTTP 202 immediately with {id, status: "processing"}.
4
go s.processGeneration(...) launches a goroutine that calls Pollinations with a 60s (or 90s) context timeout.
5
On success: saves PNG to ./storage/images/{sceneId}.png, calls UpdateCompleted() with file_path and file_url.
6
On failure: calls UpdateStatus("failed", errorCode) — ERR_03 for timeout, ERR_13 for content policy, ERR_05 for all other failures.
7
Frontend SWR polling detects status ≠ "processing" and renders the completed image or failure card.
